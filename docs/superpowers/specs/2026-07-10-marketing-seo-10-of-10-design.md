# SEO маркетингового сайта: дизайн доведения до production-уровня

## Цель

Устранить подтвержденные технические дефекты SEO в `prohelper_land` и привести публичную поисковую архитектуру в соответствие с продуктом, который уже реализован в админке и backend.

## Границы результата

Результат считается достигнутым на уровне кода, когда:

- `/blog` отдает в SSR HTML опубликованные статьи и категории, а не пустую ленту;
- мета-заголовки статей используют единый бренд `МОСТ`;
- статические URL sitemap описаны в одном источнике, ложные `lastmod` удалены, статический дубликат sitemap отсутствует;
- глобальная schema не размножает `SoftwareApplication` и предложения на каждой странице;
- удален устаревший `public/index.html` с неподтвержденным рейтингом, старой ценой и просроченной датой предложения;
- опубликованы коммерческие страницы для закупок, заявок с площадки, персонала, платежей, 1С, базы подрядчиков и AI-помощника;
- новые страницы доступны через React-маршруты, SEO registry, sitemap и внутренние ссылки;
- тесты SEO, TypeScript и ESLint подтверждают целостность изменений.

Рост позиций и трафика не является критерием завершения без данных Google Search Console, Яндекс Вебмастера и периода переобхода.

## Архитектура

### 1. Единый индекс публичных URL

Статические sitemap-маршруты хранятся в JSON, который импортируется клиентским TypeScript registry и читается серверным генератором sitemap. В статических URL не публикуется `lastmod`, если дата не поддерживается автоматически. Для статей `lastmod` берется только из `updated_at` или `published_at` API.

### 2. SSR блога

`catch-all.page.server.ts` загружает первую страницу опубликованных статей и категории только для `/blog`. Полученные данные передаются через `pageProps` в `App` и `BlogPublicPage`. Клиент инициализирует state этими данными и не повторяет первый запрос при чистом URL; фильтры, поиск и пагинация продолжают работать через текущий API-клиент. При недоступности API страница остается индексируемой, но показывает существующее понятное состояние ошибки после клиентской попытки.

### 3. Schema по типу страницы

`Organization`, `BreadcrumbList` и `WebPage` формируют базовый граф. `SoftwareApplication` добавляется только на главной и странице тарифов. `Service`, `HowTo`, `FAQPage` и `BlogPosting` остаются привязаны к содержанию конкретной страницы. Неподтвержденный `AggregateRating` не публикуется.

### 4. Новые поисковые кластеры

Новые страницы используют существующий `SeoClusterPage`, но контент хранится в отдельном `seoProductPages.ts`, чтобы не раздувать уже крупный `seoPages.ts`. Каждая страница получает уникальные intent, проблемы, автоматизации, ролевые срезы, внутренние ссылки и FAQ на основе реально существующих продуктовых контуров.

| URL | Основной intent | Продуктовое основание |
| --- | --- | --- |
| `/construction-procurement` | закупки и сравнение предложений | procurement requests, КП, выбор источника, заказ поставщику |
| `/site-requests` | заявки с площадки | site requests, статусы, потребность объекта |
| `/workforce-management` | персонал и трудозатраты | workforce management, табель, выработка |
| `/construction-payments` | платежи в строительстве | платежный реестр, бюджет, счета и НДС |
| `/one-c-integration` | интеграция стройки с 1С | 1С exchange, MDM, юридические лица и контрагенты |
| `/contractor-marketplace` | база и подбор подрядчиков | contractor marketplace в личном кабинете |
| `/ai-construction-assistant` | AI-помощник руководителя стройки | Project Pulse, отчеты, факты по закупкам, заявкам и рискам |

## Ошибки и деградация

- Сбой blog API не должен превращать `/blog` в 500.
- Некорректный ответ blog API не используется как SSR initial state.
- Неизвестный URL остается настоящим `404` с `noindex`.
- Динамический sitemap продолжает отдаваться без статей при сбое blog API.
- Новые страницы не заявляют функциональность выше maturity и фактов, зафиксированных в marketing registry и продуктовых маршрутах.

## Проверка

- Vitest: registry, sitemap, server SEO, blog SSR и рендер первой ленты.
- `npx tsc --noEmit` без сборки.
- ESLint для измененных `ts`/`tsx` файлов.
- SEO audit script по исходникам как дополнительный сигнал, с учетом его ограничений для SSR.
- Проверка `git diff --check`, отсутствия битой кодировки и устаревших упоминаний `ProHelper`, `49 900`, `AggregateRating`, `2025-12-31` в публичном SEO-контуре.
